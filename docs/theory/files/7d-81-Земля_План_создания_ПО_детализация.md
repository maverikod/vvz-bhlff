## 1. Каркас системы: какие подсистемы вообще нужны

**Причина.**
Модель Земли — это не один скрипт, а комплекс:

* единая Θ-математика,
* Земля как дефект (ядро, ПЗ, кора),
* атмосфера/океан/магнитное поле как отклик,
* внешние драйверы (Солнце, Луна, Юпитер),
* источники данных (ERA5, SST, лёд, сейсмика, магнитка…),
* модуль калибровки,
* модуль прогнозирования,
* слой сервисов и эксплуатации.

**Следствие (что делаем).**
Система делится на логические пакеты:

1. **core_theta** — общая Θ-математика (для любой планеты/звезды).
2. **earth_model** — конкретизация для Земли:

   * ядро как три тора + обмотка,
   * ПЗ как фильтр мод,
   * кора и граничные условия,
   * ветвь (n(x,\varphi)), функции (k_{\rm eff}(x,\varphi)), (n_{\rm surf}(\varphi)).
3. **drivers_solar** — драйверы от Солнца, Луны, Юпитера и др.
4. **data_sources** — единый слой доступа к данным:

   * «источник данных» как объект, а не «файл».
5. **earth_state** — текущее состояние системы («паспорт Земли»).
6. **calibration** — операторы наблюдения + оптимизация параметров.
7. **forecast** — прогнозные задачи (ячейки, циклоны, сейсмика, течения, магнитка).
8. **services** — API, CLI, задачи расписания, мониторинг.

## 2. Подсистема core_theta: единое поле и дефекты

### 2.1. Математика поля и дефекта

**Причина.**
Всё — проявление одного поля, и Земля — просто конкретный Θ-дефект. Нельзя шить код под «Землю» без общего слоя, иначе когда дойдём до Солнца/Юпитера, всё придётся переписывать.

**Следствие.**
Реализуем модуль **core_theta**:

1. Структуры данных:

   * `ThetaMode` — одна мода (\Theta_n) (параметры, частота, энергия, фаза).
   * `ThetaBranch` — ветвь (n(x,\varphi)) как функция/таблица по радиусу и широте.
   * `DefectGeometry` — абстрактное описание набора торов, обмоток и их параметров.
   * `FilterOperator` — абстрактный оператор фильтрации мод (то, что потом станет ПЗ для Земли).

2. Базовые операции:

   * построение спектра мод по заданной геометрии и ветви;
   * интегралы по радиусу/углу для вычисления плотностей, энергий и т.п.;
   * задание Q-факторов, добротностей, затухания (но без «массовых членов» в классическом виде — строго в духе теории).

3. Интерфейсы:

   * функции уровня `core_theta` **не знают слово «Земля»**, только дефект и набор мод;
   * вход — параметры дефекта и ветви (n(x,\varphi)),
   * выход — набор (\Theta_n), поля (\sigma(x,\varphi)), (\Phi(x,\varphi)) и т.п.

## 3. Подсистема earth_model: Земля как Θ-дефект

### 3.1. Геометрия ядра: три тора + обмотка

**Причина.**
Ядро задаёт главные моды, на которых строится вся остальная картина (ПЗ, кора, атмосфера, магнитка).

**Следствие.**
В модуле **earth_model** реализуем:

1. `EarthCoreGeometry`:

   * параметры трёх торов (радиусы, ориентации, углы, частоты),
   * параметризация обмотки (число витков, шаг, «фазировка» вдоль широт/долгот).

2. Функции:

   * генерация ветви (n(x,\varphi)) для ядра по этим параметрам;
   * перевод в структуры `ThetaBranch` и `DefectGeometry` из `core_theta`.

3. Связка с магнитным полем:

   * формулы, связывающие конфигурацию торов + обмотки с дипольным/многополюсным магнитным полем (без конкретных чисел, но с готовыми слотами для калибровки).

### 3.2. Переходная зона как фильтр ПЗ

**Причина.**
ПЗ задаёт фильтрацию мод ядра по радиусу/широте — задаёт, какие моды «просачиваются» к коре и в атмосферу.

**Следствие.**
Внутри **earth_model**:

1. `EarthTransitionZone`:

   * функции (k_{\rm eff}(x,\varphi)), (\sigma_{\rm crit}(x,\varphi)),
   * явный оператор фильтрации (\mathcal{F}_{\rm PZ}[\Theta_n]).

2. Интерфейс:

   * на вход — спектр мод от ядра;
   * на выход — «пропущенный» спектр, задающий структуру коры/поверхности.

### 3.3. Кора и граничные условия

**Причина.**
Кора и граница «кора–атмосфера/океан» — это граничные условия для того, что потом будет метео, сейсмикой и течениями.

**Следствие.**

1. `EarthCrustBoundary`:

   * функция (n_{\rm surf}(\varphi)) (ветвь на поверхности),
   * расчёт ожидаемых широтных ячеек:

     * зоны Hadley, Ferrel, Polar,
     * переходы 5→6, 6→7, и т.п.,
   * задание базового поля (\sigma_{\rm surf}(\varphi,\lambda)) (на долгосрочных масштабах).

2. Встроенные связи:

   * карта ячеек ⇄ будущие ячейки Hadley–Ferrel–Polar,
   * узлы и границы ячеек ⇄ будущая карта тектоники.

## 4. Подсистема drivers_solar: Солнце, Луна, планеты

**Причина.**
Подложка задаётся не только ядром Земли, но и ВБП-влиянием Солнца, Луны, Юпитера (и пр.). Парады, конфигурации, расстояния — это модуляция мод.

**Следствие.**

1. Модуль **drivers_solar**:

   * `EphemerisProvider` — абстракция для эфемерид (на первых шагах — готовые таблицы/библиотека, потом можно подключить полноценный инструмент).
   * функции вычисления:

     * (F_{\oplus\odot}(t)) — основной солнечный драйвер;
     * (F_{\oplus L}(t)) — драйвер Луны;
     * (F_{\oplus J}(t)) — вклад Юпитера и др.

2. Интерфейс с **earth_model**:

   * модуляция параметров ветви (n(x,\varphi)) и (\sigma_{\rm crit}) во времени;
   * задание сезонных смещений ячеек, приливных гармоник, прецессий.

## 5. Подсистема data_sources: источники данных

**Причина.**
Тебе важен «источник данных» как сущность, а не «файл». Внешняя реальность → оператор наблюдения → проекция на модель.

**Следствие.**

1. Модуль **data_sources**:

   * `DataSource` — базовый класс с полями:

     * идентификатор,
     * тип (атмосфера, океан, лёд, сейсмика, магнитка, тектоника, астрономия),
     * формат (GRIB/NetCDF/CSV/API…),
     * временное и пространственное разрешение,
     * статус (активен/выключен).

   * специализации:

     * `AtmosphereSource` (ERA5/NCEP),
     * `SSTSource` (OISST),
     * `SeaIceSource` (NSIDC),
     * `SeismicSource` (USGS/GCMT или выбранные каталоги),
     * `TectonicSource` (карта плит, разломов),
     * `MagneticSource` (личный набор по магнитному полю),
     * `WeatherAPISource` (публичные / коммерческие погодные API).

2. Общие функции:

   * загрузка и кеширование данных;
   * приведение к единой системе координат и времени;
   * выборка «по запросу» (например, по интервалу времени и региону).

## 6. Подсистема earth_state: «паспорт Земли»

**Причина.**
Нужен единый объект, где лежит всё состояние: параметры ядра, ПЗ, карты ячеек, калиброванные коэффициенты, текущее состояние мод.

**Следствие.**

1. Модуль **earth_state**:

   * `EarthPassport`:

     * параметры ядра (три тора, обмотка),
     * параметры ПЗ и фильтраций,
     * карта (n_{\rm surf}(\varphi)),
     * карта текущих ячеек (широты φ_H, φ_PF, их фазы),
     * параметры магнитного поля,
     * ссылки на даты/версии калибровки.

2. Хранилище:

   * реляционная/документная БД (схема «паспортов»),
   * версия состояния по времени (t) (чтобы можно было откатиться и сравнить).

## 7. Подсистема calibration: операторы наблюдения и подгонка

**Причина.**
Теория даёт формулы, но параметры (коэффициенты, ветвь, фильтры) нужно получить из данных. Нужны операторы «модель → наблюдаемое» и оптимизация «модель ↔ данные».

**Следствие.**

1. Модуль **calibration**:

   * `ObservationOperator`:

     * отдельные операторы для:

       * атмосферных полей (T, p, ветер…),
       * ячеек (широты, размещение центров),
       * циклонов (путь, интенсивность),
       * сейсмики (частота событий по регионам),
       * течений и индексов (Эль-Ниньо и т.п.),
       * магнитного поля.

   * `CostFunction`:

     * функции несоответствия модель/наблюдаемое;
     * критерии устойчивости (не только RMS, но и энерго-баланс, согласованность мод).

   * `Calibrator`:

     * процедуры оптимизации (можно начинать с простых: градиентные/стохастические/поиск по сетке);
     * сценарии:

       * калибровка ядра по магнитному полю + сейсмике;
       * калибровка ПЗ по ячейкам и потокам;
       * совместная калибровка климатических мод по ERA5/SST/лёд.

## 8. Подсистема forecast: прогнозы

**Причина.**
Практический смысл — прогнозы. Причём в той логике, которую ты сформулировал:

* тектоника задаёт **долгие тренды**,
* сейсмика подтверждает/сигнализирует,
* течения и метео конкретизируют,
* планеты подстраивают подложку.

**Следствие.**

Модуль **forecast** с задачами:

1. **Ячейки и крупномасштабная циркуляция**:

   * расчёт положения φ_H и φ_PF во времени (с учётом драйверов);
   * прогноз сезонных смещений ячеек;
   * связь с потоками атмосферы и океана.

2. **Циклогенез и ураганы**:

   * прогноз зон, где возможно рождение циклона (по модам ВБП + границам ячеек);
   * вероятностный прогноз траекторий (по фазе поля);
   * оценка изменения мощности циклона по мере движения.

3. **Сейсмическая активность**:

   * карта узлов ВБП-мод по коре;
   * прогноз вероятностей сильных событий по регионам и по окнам времени;
   * связь с движением плит и с состоянием ядра/ПЗ.

4. **Океанические явления и течения**:

   * прогноз индексов типа ЭНЮСО;
   * сдвиг границ ячеек ⇄ изменение карт течений.

5. **Магнитные события**:

   * вариации магнитного поля в связи с модами ядра и солнечной активностью;
   * прогноз геомагнитных возмущений.

## 9. Подсистема services: эксплуатация

**Причина.**
Модель и код должны жить как сервис, а не как набор скриптов.

**Следствие.**

1. API-слой:

   * REST/HTTP-интерфейс для:

     * запросов прогноза (по координате/региону/времени),
     * получения состояния «паспорта Земли»,
     * запуска процедур калибровки.

2. Планировщик:

   * регулярное обновление данных из `data_sources`,
   * периодическая коррекция состояния,
   * плановые пересчёты калибровки.

3. Мониторинг:

   * логирование ошибок/отклонений;
   * метрики производительности и качества прогнозов.

* Настроить регулярный прогон:

  * обновление данных,

## Этап 1. Математическое ядро (core_theta)

### Причина

Нужно общее Θ-ядро, независимое от «Земли», чтобы потом те же инструменты применить к Солнцу, Юпитеру, галактикам.

### Вход / выход

* **Вход:**

  * формальное ядро из теории: определение мод, ветви (n(x,\varphi)), дефектов, операторов фильтрации;
  * параметры дефекта (радиусы, частоты, топология).
* **Выход:**

  * библиотека, которая по заданной геометрии и ветви строит набор мод и основных полей (плотности/энергии и др.) — *без слов «Земля»*.

### Подробные шаги

1. **Формальная спецификация (внутри docs или core_theta/README)**

   * Чётко выписать:

     * какие типы мод есть (радиальные, угловые, комбинации),
     * какие параметры нужны каждой моде (частота, амплитуда, фаза, номер ветви, «семейство»),
     * какие базовые поля считаем (например, (\sigma), (\Phi), поток и т.п.).

2. **Структуры данных**

   * Спроектировать сущности:

     * `ThetaMode`: одна мода.
     * `ThetaBranch`: ветвь (n(x,\varphi)), реализованная как:

       * либо аналитическая функция,
       * либо таблица/интерполяция (с указанием диапазонов).
     * `DefectGeometry`: абстрактный дефект:

       * набор торов, обмоток, параметров ориентации.
     * `FilterOperator`: абстрактный оператор, который будет использоваться позднее как ПЗ.

3. **Базовые операции**

   * Функции:

     * генерация набора мод по геометрии и ветви;

## Этап 2. Земля как дефект (earth_model)

Разбиваем на три подэтапа.

### 2.1. Ядро: три тора + обмотка

#### Причина

Ядро задаёт базовый спектр мод и магнитное поле — это фундамент Земной модели.

#### Вход / выход

* **Вход:**

  * формальные параметры ядра из теории (три тора, обмотка, связи с магнитным полем).
* **Выход:**

  * модуль `EarthCoreGeometry`, который на основе параметров строит:

    * `DefectGeometry` для `core_theta`,
    * набор ветвей (n(x,\varphi)) для ядра.

#### Шаги

1. Определить формат параметров ядра:

   * радиусы трёх торов,
   * ориентации осей,
   * относительные частоты,
   * параметры обмотки (число витков, шаг, направление),
   * связи с моментом вращения Земли.

2. Реализовать `EarthCoreGeometry`:

   * конструктор, принимающий параметры;
   * метод `to_defect_geometry()` → возвращает `DefectGeometry` для `core_theta`;
   * метод(ы) генерации ветвей (n(x,\varphi)).

3. Связать ядро с магнитным полем в виде слота:

   * пока без чисел, но с каркасом функций:

     * `predict_magnetic_multipoles(core_geometry)`.

4. Написать тесты:

   * хотя бы проверка консистентности: параметры → геометрия → моды.

---

### 2.2. Переходная зона (ПЗ) как фильтр

#### Причина

ПЗ определяет, какие моды «видны» снаружи, формирует структуру коры/поверхности.

#### Вход / выход

* **Вход:**

  * спектр мод ядра (из core_theta+EarthCoreGeometry);
  * формальные формулы фильтра ПЗ из теории.
* **Выход:**

  * `EarthTransitionZone` с:

    * функциями (k_{\rm eff}(x,\varphi)), (\sigma_{\rm crit}(x,\varphi)),
    * оператором фильтрации (\mathcal{F}_{\rm PZ}).

#### Шаги

1. Определить формат описания ПЗ:

   * профиль по радиусу,
   * возможная зависимость от (\varphi),
   * параметры, которые подлежат калибровке.

2. Реализовать `EarthTransitionZone`:

   * методы:

     * `effective_conductivity(r, phi)` или аналог,
     * `sigma_crit(r, phi)`,
     * `filter_modes(theta_modes)` → отбирает/изменяет моды.

3. Связать с `core_theta`:

   * вход — массив `ThetaMode`,
   * выход — модифицированный/отрезанный спектр.

4. Тесты:

   * искусственный набор мод → проверка, что фильтр даёт ожидаемое поведение (например, отбрасывает высокочастотные или слабые моды в заданных диапазонах).

---

### 2.3. Кора и граничные условия

#### Причина

Граница коры задаёт условия для атмосферы/океана/тектоники.

#### Вход / выход

* **Вход:**

  * выход ПЗ (пропущенные моды),
  * формулы ветви (n_{\rm surf}(\varphi)).
* **Выход:**

  * `EarthCrustBoundary`:

    * карта ветви на поверхности,
    * карты ячеек,
    * условия на границе для последующих модулей.

#### Шаги

1. Реализовать функцию получения (n_{\rm surf}(\varphi)) из пропущенного спектра:

   * суммирование мод на поверхности,
   * выделение эффективной ветви.

2. Реализовать `EarthCrustBoundary`:

   * хранение (n_{\rm surf}(\varphi)),
   * функции для вычисления:

     * границ ячеек (примерно соответствующих Hadley/Ferrel/Polar),
     * зон усиления/ослабления сигнала.

3. Подготовить минимальную карту ячеек:

   * без калибровки, но согласованную с общими представлениями (30°/60° и т.п.).

4. Тесты:

   * проверка, что для заданной простой ветви (n_{\rm surf}(\varphi)) получаются ожидаемые границы ячеек.

### Готовность этапа 2

* Пакет `earth_model` умеет:

  * построить геометрию ядра,
  * пропустить моды через ПЗ,
  * получить граничные условия на поверхности (ветвь и ячейки).

## 1. Общие принципы

1. **Внутренние классы ≠ формат хранения.**
   Внутри живут **domain-классы** (например, `EarthPassport`, `EarthCoreGeometry`),
   а форматы — это **DTO/схемы** (JSON/YAML/таблицы), которые лежат в отдельном слое.

2. **Все форматы версионируются.**
   Каждый документ/таблица имеет поле `schema_version`, чтобы можно было:

   * добавлять поля,
   * менять структуру,
   * писать миграции.

3. **Форматы «читаемые человеком».**
   Для основных вещей (паспорт, конфиги) — YAML/JSON, чтобы можно было глазами посмотреть и поправить.

4. **Одна сущность — один «агрегат» хранения.**
   Пример: `EarthPassport` хранится в одном YAML/JSON-файле или в одной (или нескольких связных) таблицах БД.

---

## 2. Слой абстракции: как прячем форматы

### 2.1. Domain vs Storage

Предлагаем разделить:

* `domain/` — внутренние классы:

  * `EarthPassport`, `EarthCoreGeometry`, `EarthTransitionZone`, `EarthCrustBoundary`,
    `ThetaMode`, `ThetaBranch`, `DefectGeometry`, `DataSourceConfig`, `CalibrationScenario`, etc.
* `storage/` — адаптеры:

  * `EarthPassportRepository`,
  * `DataSourceRepository`,
  * `CalibrationRunRepository`,
  * и т.д.

**Правило:**
код в `domain/` **не знает**, как оно хранится (файлы, SQL, S3…).

### 2.2. Интерфейсы репозиториев (примерно)

На уровне идей:

```python
class EarthPassportRepository(Protocol):
    def load_latest(self) -> EarthPassport: ...
    def load_by_id(self, passport_id: str) -> EarthPassport: ...
    def save(self, passport: EarthPassport) -> str: ...
    def list_ids(self) -> list[str]: ...
```

Реализации:

* `JsonEarthPassportRepository` — хранит в `data/passports/*.json` или `*.yaml`.
* `SqlEarthPassportRepository` — хранит в БД.

Смена формата = подключили другой репозиторий.

### 2.3. Мапперы / сериализаторы

Отдельный слой:

* `EarthPassportMapper`:

  * `to_dict(passport: EarthPassport) -> dict`,
  * `from_dict(data: dict) -> EarthPassport`.

Это позволяет:

* менять формат (YAML/JSON/таблица) без изменения `EarthPassport`,
* писать миграции между версиями схемы.

## 9. Итог по слою абстракции

1. **Domain-классы**:

   * `EarthPassport`, `EarthCoreGeometry`, `EarthTransitionZone`, `EarthCrustBoundary`,
   * `ThetaMode`, `ThetaBranch`, `DefectGeometry`,
   * `DataSourceConfig`, `CalibrationScenario`, `CalibrationRun`,
   * `ForecastRequest`, `ForecastResult`.

2. **Storage-слой**:

   * интерфейсы `*Repository`,
   * реализации:

     * YAML/JSON-файлы,
     * SQL/NoSQL (позже),
   * `Mapper`-классы, превращающие `dict`/таблицы ↔ domain.

3. **Версионирование**:

   * `schema_version` в каждом документе,
   * миграции внутри storage-слоя,
   * domain-код не ломается при смене формата.

---


Ок, тогда идём системно:

1. сначала **доменные классы** (что у нас вообще есть во «внутреннем мире»),
2. потом **репозитории + мэпперы** (как это всё хранить/грузить, не привязываясь к формату).

Я буду писать каркас в виде Python-подобных классов (без реализации, только поля/методы), с комментариями по-английски, как ты просил.

## Часть 2. Репозитории и мэпперы (слой абстракции хранения)

Теперь поверх этих доменных классов — **интерфейсы репозиториев**, которые ничего не знают о том, YAML это, JSON, SQL или что-то ещё.

Для краткости буду показывать как `Protocol`/интерфейс без реализаций.

---

### 2.1. `EarthPassportRepository` и `EarthPassportMapper`

```python
from typing import Protocol


class EarthPassportRepository(Protocol):
    """
    Storage-agnostic repository for EarthPassport aggregates.
    """

    def load_latest(self) -> EarthPassport:
        """Load the latest (most recently created) EarthPassport."""
        ...

    def load_by_id(self, passport_id: str) -> EarthPassport:
        """Load EarthPassport by its ID."""
        ...

    def save(self, passport: EarthPassport) -> str:
        """
        Save EarthPassport and return its ID.
        Implementation defines how ID is generated.
        """
        ...

    def list_ids(self) -> list[str]:
        """List all stored passport IDs."""
        ...
```

Мэппер:

```python
class EarthPassportMapper:
    """
    Maps EarthPassport <-> dict representations (for JSON/YAML/DB rows).
    """

    def to_dict(self, passport: EarthPassport) -> dict:
        """Convert EarthPassport to a serializable dict."""
        ...

    def from_dict(self, data: dict) -> EarthPassport:
        """Construct EarthPassport from a dict."""
        ...
```

Реализации:

* `JsonEarthPassportRepository(EarthPassportRepository)` — использует `EarthPassportMapper`, файлы YAML/JSON.
* `SqlEarthPassportRepository(EarthPassportRepository)` — тот же мэппер, но хранит в БД.

   * API-мэпперы для запросов/ответов.

Вся логика чтения/записи/миграций сидит в storage-слое; домен не зависит от формата.
