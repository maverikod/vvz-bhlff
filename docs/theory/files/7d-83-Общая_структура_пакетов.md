## 1. Общая структура пакетов

Сразу зафиксируем целевую раскладку (потом можно детализировать):

```text
theta_earth/
  domain/              # Чистая предметная область (моды, дефект, земля, паспорта)
  infrastructure/      # Хранилища, адаптеры форматов, внешние данные
  application/         # Сервисы (EarthModelService, CalibrationService, ForecastService, ...)
  interfaces/          # API-слой: CLI, HTTP, batch-скрипты
  config/              # Конфигурация источников данных, профили калибровки
  runtime/             # Планировщики, пайплайны, фоновые задачи
  tests/               # Тесты по слоям
```

**Причина → Следствие:**

* Причина: нам нужно жёстко разделить:

  * физику и Θ-структуру (domain),
  * способ хранения/форматов (infrastructure),
  * логику сценариев (application),
  * точку входа для пользователя/систем (interfaces).
* Следствие: можно менять форматы (CSV → Parquet, SQLite → Postgres, HTTP API → локальные файлы), не трогая ни физику, ни сценарные сервисы.

## 2. Раскладка доменных классов по `domain/`

Пакет `theta_earth/domain/`:

```text
domain/
  theta_core.py          # ThetaMode, ThetaBranch, DefectGeometry
  earth_structure.py     # EarthCoreGeometry, EarthTransitionZone, EarthCrustBoundary
  field_config.py        # MagneticFieldConfig, ExternalDriverConfig (Sun/Moon/Jupiter)
  passport.py            # EarthPassport (+ вспомогательные value-объекты)
  data_sources.py        # DataSourceConfig, DataSourceStatus
  calibration.py         # CalibrationScenario, CalibrationRun, CalibrationMetrics
  forecast.py            # ForecastRequest, ForecastResult, ForecastGridSpec
```

Это уже по сути сделано логически — здесь мы **ничего не “программируем” заново**, а только фиксируем структуру файлов, чтобы все доменные сущности лежали отдельно от инфраструктуры и сценариев.

* Следствие: каждый источник заворачиваем в адаптер, который на выходе даёт **нормализованные структуры** (numpy / xarray / наши domain-модели), а репозитории оперируют только **чистыми доменными объектами**.

### 4.1. `EarthModelService`

**Роль:**
Связывает геометрию дефекта (3 тора + ПЗ + кора), параметры ВБП и конфигурацию внешних драйверов в **конкретный набор мод Земли** для заданного сценария (эпоха, конфигурация планет, наклон и т.п.).

**Зависимости:**

* `EarthPassportRepository` — чтобы брать/сохранять паспорта.
* `BaseFieldService` — чтобы получить подложку ВБП (ядро + Солнце + Луна + Юпитер).
* Возможно, `DataSourceService` — для подгонки под наблюдаемые (магнитка, сейсмика).

**Типичный интерфейс (Python-скелет):**

```python
class EarthModelService:
    def __init__(self, passport_repo, basefield_service, datasource_service):
        self.passport_repo = passport_repo
        self.basefield_service = basefield_service
        self.datasource_service = datasource_service

    def build_earth_passport(self, scenario_id: str) -> "EarthPassport":
        """
        Build full EarthPassport for given scenario (time, drivers, configs).
        """
        ...

    def update_passport_from_observations(self, passport_id: str) -> "EarthPassport":
        """
        Refine existing EarthPassport using magnetic/seismic data.
        """
        ...

    def get_core_modes(self, passport_id: str) -> list["ThetaMode"]:
        """
        Extract core-related modes from EarthPassport.
        """
        ...
```

**Причина → Следствие:**

* Причина: нам нужна **единая точка**, где понятие «модель Земли как дефекта» собирается из всех компонентов.
* Следствие: любой модуль (калибровка, прогноз) работает **через EarthPassport**, а не руками комбинирует ядро/ПЗ/кору.

### 4.2. `CalibrationService`

**Роль:**
Управляет сценариями калибровки: какие данные использовать, какие уровни (ядро, ПЗ, ячейки), какие метрики (RMSE, корреляции, энергобаланс), как сохранять результаты.

**Зависимости:**

* `CalibrationRunRepository`
* `EarthModelService` (получает/меняет паспорта)
* `DataSourceService` (достает наблюдаемые поля)
* Возможно, модуль с численными оптимизаторами (но это уже ниже уровнем).

**Типичный интерфейс:**

```python
class CalibrationService:
    def __init__(self, calibration_repo, earth_model_service, datasource_service):
        self.calibration_repo = calibration_repo
        self.earth_model_service = earth_model_service
        self.datasource_service = datasource_service

    def start_calibration(self, scenario: "CalibrationScenario") -> "CalibrationRun":
        """
        Create and start a new calibration run with given scenario.
        """
        ...

    def compute_step(self, run_id: str) -> "CalibrationRun":
        """
        Perform one iteration/step of calibration (e.g. parameter update).
        """
        ...

    def finalize_run(self, run_id: str) -> "CalibrationRun":
        """
        Mark run as finished, store final metrics and updated passport.
        """
        ...
```

**Причина → Следствие:**

* Причина: калибровка — не одноразовый скрипт, а **процесс** (итерации, перезапуски, разные сценарии).
* Следствие: нужен сервис, который понимает сценарии, статусы, логику шагов и связь с паспортом Земли.

### 4.3. `ForecastService`

**Роль:**
Строит прогнозы поверх подложки ВБП-мод и текущего EarthPassport:

* базовая сейсмическая активность (через узлы мод).

**Зависимости:**

* `EarthModelService` — берём подложку и моды.
* `DataSourceService` — для актуальных полей SST/лёд/атмосфера.
* `ForecastResultRepository` — сохранить результаты.

```python
        """
        Load previously computed forecast by id.
        """
        ...

    def list_forecasts(self, filter_params: dict) -> list["ForecastResult"]:
        """
        Search stored forecasts (by date, region, scenario, etc.).
        """
        ...
```

**Причина → Следствие:**

* Причина: прогноз — это отдельная сущность (время, область, тип прогноза, набор использованных мод).
* Следствие: нужен централизованный сервис, который:

  * принимает `ForecastRequest`,
  * вызывает нужные численные ядра,
  * сохраняет `ForecastResult`.

## I. Раздел ТЗ на программный комплекс «Θ-Земля»

Можно вставлять в `All.md` как отдельную главу «ТЗ ПО Θ-Земля».

### 1. Область применения

1.1. Программный комплекс «Θ-Земля» предназначен для:

* построения модели Земли как Θ-дефекта (три тора ядра + обмотка + ПЗ + кора),
* учёта внешних драйверов (Солнце, Луна, Юпитер) как возмущений ВБП-подложки,
* калибровки параметров модели по наблюдательным данным,
* построения прогнозов:

  * структуры широтных ячеек,
  * циклогенеза и траекторий циклонов,
  * крупномасштабных течений,
  * фоновой сейсмической активности.

1.2. Комплекс реализует **Θ-волновой подход**, не опираясь на классические уравнения механики атмосферы/грунта, а используя модовый описательный аппарат ВБП.

---

### 2. Назначение и структура комплекса

2.1. Программный комплекс логически разделён на следующие уровни:

* **Предметный уровень (domain)** — описывает:

  * Θ-моды (`ThetaMode`, `ThetaBranch`, `DefectGeometry`),
  * структуру Земли (`EarthCoreGeometry`, `EarthTransitionZone`, `EarthCrustBoundary`),
  * конфигурацию магнитного поля и внешних драйверов (`MagneticFieldConfig`, внешние драйверы),
  * паспорт Земли (`EarthPassport`),
  * сценарии калибровки (`CalibrationScenario`, `CalibrationRun`),
  * сущности прогнозов (`ForecastRequest`, `ForecastResult`),
  * конфигурации источников данных (`DataSourceConfig`, `DataSourceStatus`).

* **Инфраструктурный уровень (infrastructure)** — реализует:

  * хранилища паспортов, калибровок и прогнозов (репозитории),
  * адаптеры источников данных (ERA5, SST, лёд, сейсмика, магнитка),
  * мэпперы между внутренними моделями и внешними форматами (dict, JSON, таблицы).

* **Сервисный уровень (application)** — реализует:

  * сборку и ведение модели Земли (`EarthModelService`),
  * процессы калибровки (`CalibrationService`),
  * построение прогнозов (`ForecastService`),
  * доступ к источникам данных (`DataSourceService`),
  * управление паспортами (`PassportService`),
  * расчёт Θ-подложки (Земля+Солнце+Луна+Юпитер) (`BaseFieldService`).

* **Интерфейсный уровень (interfaces)** — предоставляет:

  * HTTP API для внешних систем,
  * CLI-интерфейс для экспериментов и пакетного запуска.

### 3. Функциональные требования (по блокам)

#### 3.1. Блок «Модель Земли как Θ-дефект»

3.1.1. Система должна обеспечивать создание новых паспортов Земли
на основании входной конфигурации (геометрия ядра, параметры ПЗ, коры, внешних драйверов).

* Реализуется методами `EarthModelService`:

  * `create_passport(config: dict)`,
  * `save_passport(passport)`.

3.1.2. Система должна обеспечивать построение полного паспорта Земли
для заданного сценария (эпоха, конфигурация планет, внешние драйверы ВБП).

* Реализуется:

  * `build_earth_passport(scenario_id: str)`.

3.1.3. Система должна поддерживать пере-построение паспорта
при изменении конфигурации внешних драйверов (время, положение Солнца, Луны, Юпитера).

* Реализуется:

  * `rebuild_passport_with_new_drivers(passport_id: str, new_driver_config: dict)`.

3.1.4. Система должна предоставлять доступ к модам, относящимся к:

* ядру (основные моды, связанные с магнитным полем и сейсмикой),

* переходной зоне (моды-фильтры, задающие ячейки),

* коре (граничные условия).

* Реализуется:

  * `get_core_modes(passport_id: str)`,
  * `get_transition_zone_modes(passport_id: str)`,
  * `summarize_modes(passport_id: str)`.

---

#### 3.2. Блок «Подложка ВБП: Земля+Солнце+Луна+Юпитер»

3.2.1. Система должна вычислять конфигурацию Θ-подложки
для заданного момента времени с учётом вклада Земли, Солнца, Луны и Юпитера.

* Реализуется в `BaseFieldService`:

  * `compute_basefield(time_point)`.

3.2.2. Система должна предоставлять результаты Θ-подложки для дальнейшего использования
при построении паспортов и прогнозов.

* Используется `EarthModelService.build_earth_passport` и `ForecastService.run_forecast`.

---

#### 3.3. Блок «Калибровка по наблюдательным данным»

3.3.1. Система должна поддерживать создание сценариев калибровки, включающих:

* перечисление калибруемых параметров,

* перечень используемых наборов данных (магнитное поле, сейсмика, ERA5, SST, лёд и т.п.),

* набор метрик (RMSE, корреляция, энергетический баланс).

* Реализуется в `CalibrationService`:

  * `create_scenario(scenario: CalibrationScenario)`.

3.3.2. Система должна поддерживать запуск калибровочного прогона
на основе выбранного сценария и паспорта Земли.

* `start_calibration(scenario_id: str, passport_id: str)`.

3.3.3. Система должна обеспечивать пошаговое выполнение калибровки
с возможностью контроля и остановки:

* `compute_step(run_id: str)`,
* `run_until_converged(run_id: str, max_steps: int | None)`.

3.3.4. Система должна сохранять и предоставлять метрики калибровки:

* `get_run_metrics(run_id: str)`,
* `compare_runs(run_id_a: str, run_id_b: str)`.

3.3.5. Система должна обновлять паспорт Земли
по результатам калибровки (магнитка, сейсмика, ERA5/SST/лёд) и фиксировать версионность.

* Связка `CalibrationService.finalize_run` + `EarthModelService.update_passport_from_observations`.

---

#### 3.4. Блок «Прогнозы (ячейки, циклоны, течения, сейсмика)»

3.4.1. Система должна строить прогноз конфигурации широтных ячеек
(границы φ_H, φ_PF, их сезонные смещения) для заданного временного окна и области.

* Реализуется в `ForecastService`:

  * `forecast_cells(request: ForecastRequest)`.

3.4.2. Система должна строить прогноз циклогенеза и траекторий циклонов
на основе мод ВБП-подложки и актуальных полей SST/льда/атмосферы.

* `forecast_cyclogenesis(request)`.

3.4.3. Система должна строить прогноз крупномасштабных течений
(джеты, течения) с учётом Θ-ячеек и морской поверхности.

* `forecast_flows(request)`.

3.4.4. Система должна оценивать фоновую сейсмическую активность
на основе распределения Θ-мод и узлов.

* `forecast_seismic_background(request)`.

3.4.5. Система должна сохранять результаты каждого прогноза
с указанием:

* id паспорта Земли,

* использованных источников данных,

* временного интервала и области.

* `run_forecast(request) -> ForecastResult`,

* `get_forecast(forecast_id)`,

* `list_forecasts(filter_params)`.

---

#### 3.5. Блок «Источники данных»

3.5.1. Система должна обеспечивать единый интерфейс доступа
ко всем источникам данных (ERA5, SST, лёд, сейсмика, магнитка).

* Реализуется в `DataSourceService`:

  * `fetch_field(source_id, field_name, time_range, region)`.

3.5.2. Система должна вести учёт статуса источников
(наличие данных, дата последнего обновления, ошибки).

* `get_status(source_id)`.

3.5.3. Система должна позволять конфигурировать источники данных
(пути к файлам, URL сервисов, параметры аутентификации и пр.).

* Описывается через `DataSourceConfigRepository` и `DataSourceService`.

#### 1.1. Паспорт Земли

* `GET /api/passports`

  * Назначение: получить список паспортов Земли (с фильтрами по сценарию, дате создания и т.п.).
  * Вызывает: `EarthModelService` + репозиторий паспортов.

  * Тело: JSON с high-level config (геометрия, ПЗ, кора, драйверы).

  * Вызывает: `create_passport` → `save_passport`.

  * Назначение: получить полный паспорт.

  * Назначение: создать копию паспорта с новым именем/меткой.

* `POST /api/passports/{passport_id}/rebuild-drivers`

  * Тело: новая конфигурация драйверов.
  * Назначение: перестроить паспорт под новые внешние драйверы.
  * Вызывает: `rebuild_passport_with_new_drivers`.

* `POST /api/passports/{passport_id}/update-from-observations`

  * Тело: описание используемых наборов данных и периода.

   * HTTP-API для внешних систем;
   * CLI-утилиту для исследовательской работы и пакетных сценариев.

---

### **80.2. Функциональные требования: модель Земли как дефект**

80.2.1. Комплекс должен обеспечивать **создание нового паспорта Земли** на основании высокоуровневой конфигурации:

* геометрия ядра и трёх торов;
* параметры обмотки;
* параметры переходной зоны (фильтра);
* параметры коры (граничные условия);
* конфигурация внешних драйверов (Солнце, Луна, Юпитер).

Реализация:
`EarthModelService.create_passport(config)`, `EarthModelService.save_passport(passport)`.

80.2.2. Комплекс должен обеспечивать **построение полного паспорта Земли** для заданного сценария (эпоха, положение планет, конфигурация драйверов).

Реализация:
`EarthModelService.build_earth_passport(scenario_id)` с использованием `BaseFieldService.compute_basefield(time_point)`.

80.2.3. Комплекс должен поддерживать **перестройку паспорта** при изменении внешних драйверов (другая эпоха, другая конфигурация Солнце–Луна–Юпитер).

Реализация:
`EarthModelService.rebuild_passport_with_new_drivers(passport_id, new_driver_config)`.

80.2.4. Комплекс должен предоставлять доступ к Θ-модам, относящимся к:

* ядру (магнитное поле, сейсмика),
* переходной зоне (фильтрация мод и ячейки),
* ячейкам (Hadley/Ferrel/Polar).

Реализация:
`EarthModelService.get_core_modes(passport_id)`,
`EarthModelService.get_transition_zone_modes(passport_id)`,
`EarthModelService.get_cell_modes(passport_id)`,
`EarthModelService.summarize_modes(passport_id)`.

---

### **80.3. Подложка ВБП: Земля + Солнце + Луна + Юпитер**

80.3.1. Комплекс должен вычислять **конфигурацию Θ-подложки** для заданного момента времени с учётом вклада Земли, Солнца, Луны, Юпитера.

Реализация:
`BaseFieldService.compute_basefield(time_point)`.

80.3.2. Данные Θ-подложки должны использоваться при:

* построении паспорта Земли (`EarthModelService.build_earth_passport`),
* построении прогнозов (`ForecastService.run_forecast`).

---

### **80.4. Калибровка модели по наблюдениям**

80.4.1. Комплекс должен поддерживать **создание сценариев калибровки**, включающих:

* список калибруемых параметров (ядро, ПЗ, ячейки и т.п.);
* перечень используемых наборов данных (магнитка, сейсмика, ERA5, SST, лёд);
* набор метрик (RMSE по слоям, корреляции, энергетический баланс).

Реализация:
`CalibrationService.create_scenario(scenario)`.

80.4.2. Комплекс должен поддерживать **запуск калибровочных прогонов** на основе выбранного сценария и паспорта Земли.

Реализация:
`CalibrationService.start_calibration(scenario_id, passport_id)`.

80.4.3. Комплекс должен обеспечивать **пошаговое выполнение калибровки** с возможностью итеративного обновления параметров.

Реализация:
`CalibrationService.compute_step(run_id)`,
`CalibrationService.run_until_converged(run_id, max_steps)`.

80.4.4. Комплекс должен сохранять и предоставлять **метрики калибровки**, а также сравнение различных прогонов.

Реализация:
`CalibrationService.get_run_metrics(run_id)`,
`CalibrationService.compare_runs(run_id_a, run_id_b)`.

80.4.5. По завершении калибровки комплекс должен обновлять и версияционировать паспорт Земли.

Реализация:
`CalibrationService.finalize_run(run_id)` → обновлённый `EarthPassport` через `EarthModelService`.

---

### **80.5. Прогнозы: ячейки, циклоны, течения, сейсмика**

80.5.1. Комплекс должен строить прогноз **конфигурации широтных ячеек** (φ_H, φ_PF и их сезонные сдвиги) для заданного временного окна и области.

Реализация:
`ForecastService.forecast_cells(request)`.

80.5.2. Комплекс должен строить прогноз **циклогенеза и траекторий циклонов** на основе мод ВБП и актуальных полей SST/льда/атмосферы.

Реализация:
`ForecastService.forecast_cyclogenesis(request)`.

80.5.3. Комплекс должен строить прогноз **крупномасштабных течений** (джеты, течения) с учётом Θ-ячейчной структуры и морской поверхности.

Реализация:
`ForecastService.forecast_flows(request)`.

80.5.4. Комплекс должен оценивать **фоновую сейсмическую активность** на основе расположения узлов Θ-мод и их энергетики.

Реализация:
`ForecastService.forecast_seismic_background(request)`.

80.5.5. Каждый прогноз должен сохраняться с явным указанием:

* паспорта Земли,
* источников данных,
* типа прогноза,
* временного интервала и области.

Реализация:
`ForecastService.run_forecast(request)` + `ForecastResultRepository`.

---

### **80.6. Источники данных и их управление**

80.6.1. Комплекс должен обеспечивать **единый интерфейс доступа** ко всем источникам данных (ERA5, SST, лёд, сейсмика, магнитка, др.).

Реализация:
`DataSourceService.fetch_field(source_id, field_name, time_range, region)`.

80.6.2. Комплекс должен вести учёт **статуса источников** (наличие данных, дата обновления, ошибки).

Реализация:
`DataSourceService.get_status(source_id)`.

80.6.3. Конфигурация источников (пути, URL, параметры доступа) должна храниться и управляться централизованно.

Реализация:
`DataSourceConfigRepository` + `DataSourceService`.

---

### **80.7. Интерфейсы: HTTP-API и CLI**

80.7.1. Должен быть реализован HTTP-API, предоставляющий доступ ко всем функциям комплекса:

* управление паспортами (`/api/passports/...`);
* управление калибровкой (`/api/calibration/...`);
* запуск и получение прогнозов (`/api/forecasts/...`);
* управление источниками данных (`/api/datasources/...`).

80.7.2. Должна быть реализована CLI-утилита (`earthctl`) с командами:

* `earthctl passport ...` — операции над паспортами;
* `earthctl calib ...` — операции калибровки;
* `earthctl forecast ...` — запуск и просмотр прогнозов;
* `earthctl datasource ...` — управление источниками данных.
